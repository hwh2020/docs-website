#### 解析

解析是RAG流程的“地基”，其质量直接决定了后续检索和生成的上限。如果解析环节出现错误，后续再多优化也难以弥补。

##### 常见的错误

- 阅读顺序错乱

  文档中的分栏、目录、页眉页脚等元素可能导致段落顺序被打乱，进而造成检索时上下文断裂。

- 表格结构丢失

  表格的列名、合并单元格、行列关系等结构信息在解析后可能丢失或错位，导致关键数据无法被准确召回。

- 噪声污染

  页码、免责声明、重复的页眉页脚等非内容信息被错误地注入，使检索结果包含大量无关内容。

- 图文断裂

  图片丢失、位置错乱或无法引用，导致多模态信息无法有效利用。

##### 解析难度分级

- 1星 纯文本 / Markdown

  结构清晰，直接使用基础的解析器

- 2星 Word / EXCEL / PPT

  格式丢失，表格错乱

- 3星 PDF （文字型）

  分栏问题，页眉页脚干扰

- 4星 PDF （扫描型）

  OCR精度波动，版面复杂

##### 技术路线

- 传统 Pipeline OCR

  - 流程

    多个小模型串联组成，包括：

    - 版面分区
    - OCR 识别
    - 表格 / 公式专项处理
    - 顺序还原
    - 导出结构化数据

  - 优点

    每个环节相对独立，可控性强

    易于调试和解释

  - 缺点

    链路较长，误差可能累积叠加

    对复杂版面的处理能力有限

- VLM / 多模态端到端解析

  - 流程

    利用视觉语言模型或多模态模型，以更一体化的方式

    - 版面理解
    - 元素识别
    - 阅读顺序还原

  - 优点

    处理复杂版面时更省心

    更好地理解文档整体结构和语义

  - 缺点

    成本较高

    模型一致性与部署稳定性需额外关注

业界常用：MinerU 2.5、HunyuanOCR、PaddleOCR



#### 清洗

数据清洗在RAG流程中扮演着 “效果放大器” 的角色，目的是去噪、统一、可结构化、可追溯

##### 四大目标

==去噪、统一、结构化、可追溯==

- 页眉页脚 / 页码 / 水印去噪

  移除文档中非核心内容的干扰信息，避免其被误认为是正文

- 断行与连字符修复

  修复因换行导致的词语断裂，确保词语的完整性

- 重复段落去重

  识别并删除文档中重复出现的段落，如免责声明、重复的目录等。

- 表格“保列名/保行结构”

  确保表格数据在解析后仍能保持其原有的列名和行结构

- 符号规范化

  统一全角 / 半角字符、特殊字符和单位的表示方式，提高文本一致性

- 标题层级修复

  修复或识别文档中的标题层级结构，为结构分块打下基础。

高质量的清洗能够显著提升后续分块、召回和引用的稳定性与准确性



#### 分块

它决定了检索的颗粒度与引用内容的可用性

根本原因：大模型的上下文窗口有限，长文本需要被拆分成“可召回的基本单位”

##### 关键参数

- 块大小 Chunk Size

  太大：包含过多无关信息、增加噪声、降低检索精度

  太小：导致语义断裂、概念被切分成多个不连贯的块

- 重叠 Overlap

  设置重叠：缓解语义断裂问题，提高上下文连贯性

  增加成本：重叠会增加数据冗余、提高存储和检索成本

- 策略 Strategy

  选择策略：不同文档类型和内容结构需要不同的分块策略

  关键旋钮：选择合适的策略往往比单纯调整块大小更关键

##### 分块策略

- 固定长度

  按照预设的token或字符数量进行切割

  效果不稳定，易在语义中间切断

- 句子滑窗

  以句子为基本单位，允许相邻句子之间有重叠

  避免切半句，符合人类阅读习惯

  有可能拆分太细了，检索召回的时候没有被完整召回，存在回答上的一些偏差。

- 结构化分块

  根据文档结构（标题、段落、列表等）进行切割（企业文档中最常用，保持语义完整）

- 语义分块

  利用语义模型识别文本的语义边界进行切割

  成本高，可控性难

- 父子分块

  子块 - 较小的文本片段，用于提高检索的命中率

  父块 - 更大的上下文单元，提高完整的背景信息

  在不设置大 overlap 的情况下，有效补齐了上下文，提高生成答案的质量。

元数据补强：chunk_id、page(页码）、paragraph（段落号）、section_path（章节路径）来源ID

进一步还有pre_id, next_id 等等,支持相邻块扩展和跳转定位。



#### 向量化

将文本信息转换为高维向量空间中的数值表示，这些向量能够捕捉文本的语义信息。

相似度衡量：相似度尤其是余弦相似度，可以理解为衡量两个文本在语义空间中“方向像不像”的程度。方向越一致，语义越相似。

##### 常见向量模型

- OpenAI 

  text-embedding-3-large

- BAAI

  bge-m3

- Qwen

  Qwen3-Embedding-8B

##### 检索的局限

由于语义相似度不足以弥补字面上的巨大差异，纯向量检索的TopK 结果可能不稳定，甚至无法命中相关文档。本质问题：语义相似不等于字面匹配。

解决方案：

#### 混合检索（Hybrid）

混合检索结合向量检索和关键词检索的优势，通过以下方式弥补纯向量检索的不足：

- 别名 / 同义词机制

- 元数据过滤

  通过类似部门、时间、文档类型等元数据进行预过滤，缩小检索范围

- 倒排索引（BM25）

  擅长处理字面匹配场景，适用于编号、缩写等。精确匹配需求场景

- Query Rewrite

  将口语化查询重写为更利于检索的规范化问法

- Multi-Query

  并行发送多个改写后的查询，合并候选结果



#### 重排

初步召回的TopK结果只是一个候选集，可能包含大量“差不多”但并非最相关的文档

重排的目的是从候选集中找出真正最相关的证据

重排模型能够捕捉Query和文档之间的细粒度交互，提供更精细的排序

重排模型类型

- Cross-Encoder

  联合编码，更精确，计算密集

- Bi-Encoder

  分别编码，速度快，精度略低



#### 上下文组装

##### 四个核心步骤

- 去重

  消除检索结果中重复或高度相似的片段，避免信息冗余

- 排序

  根据相关性，重要性或文档结构对证据进行逻辑排序

- 压缩

  对上下文进行精简，适应大模型的上下文窗口限制

- 标注来源

  为每个证据片段标注原始出处，确保生成答案的可追溯性



#### 图文混排处理

在RAG中， 处理图文混排文档的核心思想是将图片作为可引用的资产进行处理

##### 流程

- 解析

  从文档中抽取图片，存储到对象存储服务（OSS）中，并生成对于的HTTP URL

  注意：图片链接的命名要有一定的业务含义

- 回填

  在解析后的文本内容中，按照图片原有的位置插入图片链接（如 `![](rul)`或标签）

- 检索

  当检索系统命中包含图片链接的文本块时，将图片链接也会作为证据的一部分被召回

- 生成

  在向大模型发送提示词时，明确要求模型“不要改写 URL，原样输出”

##### 场景

- 展示型

  前端渲染 - 前端界面接受到包含图片URL的回答后，直接渲染图片，实现图文混排的展示。

- 理解型

  使用多模态模型对每个图片生成文本的描述（caption），把描述也插入到解析好的Markdown文本中，实际问答的时候，图片的内容也可以被检索到，实现了搜图的效果。



#### 评估闭环

评估方法

- 离线评估

  上线前：构建黄金测试集进行评估，为系统优化提供方向

- 在线评估

  上线后：收集用户在实际使用中的反馈信号，真实反映系统表现并持续迭代

用户反馈信息

显示信号

- 用户对回答满意度的主动反馈
- 引用点击： 用户点击引用链接查看原文
- 纠错反馈：用户提交的错误报告或修正建议

隐式信号

- 改写重问：用户对当前回答不满意，尝试改写问题
- 同问多次：用户短时间内多次提出相同问题
- 转人功率：用户从RAG系统转向人工客服的比率
- 复制率：用户复制RAG回答内容的频率
- 停留时长：用户在回答页面停留的时间

通过持续收集和分析用户反馈信号，形成完整评估闭环，不断优化RAG系统性能。

### Agent

使用Agent的四个情况:

- 目标模糊

  能说清"要什么结果",但说不清"怎么做",例如: "帮我分析销售下降的原因"

- 路径不确定

  下一步取决于上一步结果,需要边做边决定,例如:数据探索,根因分析

- 环境多变

  依赖多系统 / 多数据源, 随时可能超时,报错,缺数据

- 探索试错

  需要先尝试几条路线,再收敛到更优方案,例如:策略制定,方案比选

护栏

- 权限控制

  能访问哪些系统?能不能写库 / 发消息?

  白名单机制,只读权限,敏感操作审批

- 可观测审计

  每一步做了什么?用了什么工具?证据链在哪里?

  全链路日志,决策轨迹记录, 可回放

- 成本与限额

  调用次数/ 时长/ Token/ 预算上限

  硬性上限,超限自动终止,成本预警

- 人工兜底

  低置信度/高风险场景转人工

  关键节点挂起审批, 可回滚, 人工确认
